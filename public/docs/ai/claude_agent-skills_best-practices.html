Skill authoring best practices

좋은 스킬은 간결하고, 잘 구조화되어 있으며, 실제 사용 사례로 테스트되었습니다. 이 가이드는 Claude가 효과적으로 발견하고 사용할 수 있는 스킬을 작성하는 데 도움이 되는 실용적인 작성 결정 사항을 제공합니다.

좋은 스킬은 간결하고, 잘 구조화되어 있으며, 실제 사용 사례로 테스트되었습니다. 이 가이드는 Claude가 효과적으로 발견하고 사용할 수 있는 스킬을 작성하는 데 도움이 되는 실용적인 작성 결정 사항을 제공합니다.

스킬의 작동 방식에 대한 개념적 배경은 스킬 개요를 참조하세요.

핵심 원칙

간결함이 핵심입니다

컨텍스트 윈도우는 공공재입니다. 여러분의 스킬은 Claude가 알아야 할 다른 모든 것과 컨텍스트 윈도우를 공유합니다:

  • 시스템 프롬프트
  • 대화 기록
  • 다른 스킬의 메타데이터
  • 실제 요청 내용

스킬의 모든 토큰이 즉각적인 비용을 발생시키는 것은 아닙니다. 시작 시 모든 스킬의 메타데이터(이름과 설명)만 미리 로드됩니다. Claude는 스킬이 관련성이 있을 때만 SKILL.md를 읽고, 필요에 따라 추가 파일만 읽습니다. 그러나 SKILL.md에서 간결함은 여전히 중요합니다: Claude가 로드한 후에는 모든 토큰이 대화 기록 및 기타 컨텍스트와 경쟁하기 때문입니다.

기본 가정: Claude는 이미 매우 똑똑합니다. Claude가 이미 가지고 있지 않은 컨텍스트만 추가하세요. 각 정보에 대해 다음을 질문하세요:
  • "Claude가 정말로 이 설명이 필요한가?"
  • "Claude가 이것을 알고 있다고 가정할 수 있는가?"
  • "이 단락이 토큰 비용을 정당화하는가?"

좋은 예시: 간결함 (약 50 토큰)

## PDF 텍스트 추출

텍스트 추출에는 pdfplumber를 사용하세요:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

나쁜 예시: 너무 장황함 (약 150 토큰)

## PDF 텍스트 추출

PDF(Portable Document Format) 파일은 텍스트, 이미지 및 기타 콘텐츠를 포함하는 
일반적인 파일 형식입니다. PDF에서 텍스트를 추출하려면 라이브러리를 사용해야 합니다. 
PDF 처리에 사용할 수 있는 라이브러리가 많지만, 대부분의 경우를 잘 처리하고 
사용하기 쉬운 pdfplumber를 추천합니다. 먼저 pip를 사용하여 설치해야 합니다. 
그런 다음 아래 코드를 사용할 수 있습니다...

간결한 버전은 Claude가 PDF가 무엇인지, 라이브러리가 어떻게 작동하는지 알고 있다고 가정합니다.

적절한 자유도 설정

작업의 취약성과 변동성에 맞게 구체성 수준을 조정하세요.

높은 자유도 (텍스트 기반 지침)

다음의 경우 사용:

  • 여러 접근 방식이 유효한 경우
  • 결정이 컨텍스트에 따라 달라지는 경우
  • 휴리스틱이 접근 방식을 안내하는 경우
## 코드 리뷰 프로세스

1. 코드 구조와 조직을 분석합니다
2. 잠재적인 버그나 엣지 케이스를 확인합니다
3. 가독성 및 유지보수성 개선 사항을 제안합니다
4. 프로젝트 규칙 준수를 확인합니다

중간 자유도 (매개변수가 있는 의사코드 또는 스크립트)

다음의 경우 사용:

  • 선호하는 패턴이 존재하는 경우
  • 일부 변형이 허용되는 경우
  • 구성이 동작에 영향을 미치는 경우
## 보고서 생성

이 템플릿을 사용하고 필요에 따라 사용자 정의하세요:

```python
def generate_report(data, format="markdown", include_charts=True):
    # 데이터 처리
    # 지정된 형식으로 출력 생성
    # 선택적으로 시각화 포함
```

낮은 자유도 (구체적인 스크립트, 매개변수 거의 없음)

다음의 경우 사용:

  • 작업이 취약하고 오류가 발생하기 쉬운 경우
  • 일관성이 중요한 경우
  • 특정 순서를 따라야 하는 경우
## 데이터베이스 마이그레이션

정확히 이 스크립트를 실행하세요:

```bash
python scripts/migrate.py --verify --backup
```

명령을 수정하거나 추가 플래그를 추가하지 마세요.
비유: Claude를 경로를 탐색하는 로봇으로 생각하세요:
  • 양쪽에 절벽이 있는 좁은 다리: 안전한 길은 하나뿐입니다. 구체적인 가드레일과 정확한 지침을 제공하세요 (낮은 자유도). 예: 정확한 순서로 실행되어야 하는 데이터베이스 마이그레이션.
  • 장애물이 없는 탁 트인 들판: 많은 경로가 성공으로 이어집니다. 일반적인 방향을 제시하고 Claude가 최선의 경로를 찾도록 신뢰하세요 (높은 자유도). 예: 컨텍스트가 최선의 접근 방식을 결정하는 코드 리뷰.

사용할 모든 모델로 테스트

스킬은 모델에 대한 추가 기능으로 작동하므로 효과는 기본 모델에 따라 달라집니다. 사용하려는 모든 모델로 스킬을 테스트하세요.

모델별 테스트 고려사항:

  • Claude Haiku (빠르고 경제적): 스킬이 충분한 지침을 제공하는가?
  • Claude Sonnet (균형): 스킬이 명확하고 효율적인가?
  • Claude Opus (강력한 추론): 스킬이 지나치게 설명하지 않는가?

Opus에 완벽하게 작동하는 것이 Haiku에는 더 많은 세부 사항이 필요할 수 있습니다. 여러 모델에서 스킬을 사용할 계획이라면 모두에게 잘 작동하는 지침을 목표로 하세요.

스킬 구조

참고: SKILL.md frontmatter는 두 개의 필드가 필요합니다:
  • name: 최대 64자, 소문자, 숫자, 하이픈만 포함, XML 태그 금지, 예약어("anthropic", "claude") 금지
  • description: 비어 있지 않음, 최대 1024자, XML 태그 금지, 스킬의 기능과 사용 시기를 설명해야 함

전체 스킬 구조 세부 정보는 스킬 개요를 참조하세요.

명명 규칙

일관된 명명 패턴을 사용하면 스킬을 더 쉽게 참조하고 논의할 수 있습니다. 스킬 이름에는 동명사 형태(동사 + -ing)를 사용하는 것이 좋습니다. 이는 스킬이 제공하는 활동이나 기능을 명확하게 설명하기 때문입니다.

name 필드는 소문자, 숫자, 하이픈만 사용해야 함을 기억하세요.

좋은 명명 예시 (동명사 형태):

  • processing-pdfs
  • analyzing-spreadsheets
  • managing-databases
  • testing-code
  • writing-documentation

피해야 할 것:

  • 모호한 이름: helper, utils, tools
  • 지나치게 일반적: documents, data, files
  • 예약어: anthropic-helper, claude-tools
  • 스킬 컬렉션 내에서 일관성 없는 패턴

효과적인 설명 작성

description 필드는 스킬 발견을 가능하게 하며 스킬이 무엇을 하는지와 언제 사용해야 하는지를 모두 포함해야 합니다.

항상 3인칭으로 작성하세요. 설명은 시스템 프롬프트에 주입되며, 일관성 없는 관점은 발견 문제를 일으킬 수 있습니다.
  • 좋음: "Excel 파일을 처리하고 보고서를 생성합니다"
  • 피할 것: "Excel 파일 처리를 도와드릴 수 있습니다"
  • 피할 것: "이것을 사용하여 Excel 파일을 처리할 수 있습니다"

구체적이고 핵심 용어를 포함하세요

각 스킬에는 정확히 하나의 설명 필드가 있습니다. 설명은 스킬 선택에 중요합니다: Claude는 이를 사용하여 잠재적으로 100개 이상의 사용 가능한 스킬에서 올바른 스킬을 선택합니다. 설명은 Claude가 이 스킬을 선택해야 하는 시기를 알 수 있도록 충분한 세부 정보를 제공해야 하며, SKILL.md의 나머지 부분은 구현 세부 정보를 제공합니다.

효과적인 예시:

description: PDF 파일에서 텍스트와 테이블을 추출하고, 양식을 작성하고, 문서를 병합합니다. 
PDF 파일을 작업할 때 또는 사용자가 PDF, 양식 또는 문서 추출을 언급할 때 사용하세요.
description: Excel 스프레드시트를 분석하고, 피벗 테이블을 만들고, 차트를 생성합니다. 
Excel 파일, 스프레드시트, 표 형식 데이터 또는 .xlsx 파일을 분석할 때 사용하세요.

피해야 할 모호한 설명:

description: 문서를 도와줍니다
description: 데이터를 처리합니다

워크플로우와 피드백 루프

복잡한 작업에는 워크플로우 사용

복잡한 작업을 명확하고 순차적인 단계로 나누세요. 특히 복잡한 워크플로우의 경우 Claude가 응답에 복사하고 진행하면서 체크할 수 있는 체크리스트를 제공하세요.

예시 1: 연구 종합 워크플로우 (코드가 없는 스킬)

## 연구 종합 워크플로우

이 체크리스트를 복사하고 진행 상황을 추적하세요:

```
연구 진행 상황:
- [ ] 1단계: 모든 소스 문서 읽기
- [ ] 2단계: 주요 주제 식별
- [ ] 3단계: 주장 교차 참조
- [ ] 4단계: 구조화된 요약 작성
- [ ] 5단계: 인용 확인
```

**1단계: 모든 소스 문서 읽기**

sources/ 디렉토리의 각 문서를 검토하세요. 주요 논점과 뒷받침 증거를 기록하세요.

**2단계: 주요 주제 식별**

소스 전반에 걸친 패턴을 찾으세요. 반복적으로 나타나는 주제는 무엇인가요? 
소스가 동의하거나 동의하지 않는 부분은 어디인가요?

**3단계: 주장 교차 참조**

각 주요 주장에 대해 소스 자료에 나타나는지 확인하세요. 각 요점을 뒷받침하는 
소스를 기록하세요.

**4단계: 구조화된 요약 작성**

주제별로 발견 사항을 정리하세요. 포함 사항:
- 주요 주장
- 소스의 뒷받침 증거
- 충돌하는 관점 (있는 경우)

**5단계: 인용 확인**

모든 주장이 올바른 소스 문서를 참조하는지 확인하세요. 인용이 불완전하면 
3단계로 돌아가세요.

피드백 루프 구현

일반적인 패턴: 유효성 검사기 실행 → 오류 수정 → 반복

이 패턴은 출력 품질을 크게 향상시킵니다.

예시 1: 스타일 가이드 준수 (코드가 없는 스킬)

## 콘텐츠 검토 프로세스

1. STYLE_GUIDE.md의 지침에 따라 콘텐츠를 작성합니다
2. 체크리스트에 따라 검토합니다:
   - 용어 일관성 확인
   - 예제가 표준 형식을 따르는지 확인
   - 필수 섹션이 모두 있는지 확인
3. 문제가 발견되면:
   - 특정 섹션 참조와 함께 각 문제를 기록
   - 콘텐츠를 수정
   - 체크리스트를 다시 검토
4. 모든 요구 사항이 충족될 때만 진행
5. 문서를 최종 확정하고 저장

예시 2: 문서 편집 프로세스 (코드가 있는 스킬)

## 문서 편집 프로세스

1. word/document.xml을 편집합니다
2. **즉시 유효성 검사**: python ooxml/scripts/validate.py unpacked_dir/
3. 유효성 검사가 실패하면:
   - 오류 메시지를 주의 깊게 검토
   - XML의 문제를 수정
   - 유효성 검사를 다시 실행
4. **유효성 검사가 통과할 때만 진행**
5. 재빌드: python ooxml/scripts/pack.py unpacked_dir/ output.docx
6. 출력 문서를 테스트

유효성 검사 루프는 오류를 조기에 포착합니다.

일관된 용어 사용

하나의 용어를 선택하고 스킬 전체에서 사용하세요:

좋음 - 일관성 있음:

  • 항상 "API 엔드포인트"
  • 항상 "필드"
  • 항상 "추출"

나쁨 - 일관성 없음:

  • "API 엔드포인트", "URL", "API 경로", "경로" 혼용
  • "필드", "상자", "요소", "컨트롤" 혼용
  • "추출", "가져오기", "얻기", "검색" 혼용

일관성은 Claude가 지침을 이해하고 따르는 데 도움이 됩니다.

평가 및 반복

먼저 평가를 구축하세요

광범위한 문서를 작성하기 전에 평가를 생성하세요. 이렇게 하면 스킬이 상상한 문제가 아닌 실제 문제를 해결하도록 보장합니다.

평가 주도 개발:

  1. 격차 식별: 스킬 없이 대표적인 작업에서 Claude를 실행하세요. 특정 실패 또는 누락된 컨텍스트를 문서화하세요
  2. 평가 생성: 이러한 격차를 테스트하는 세 가지 시나리오를 구축하세요
  3. 기준선 설정: 스킬 없이 Claude의 성능을 측정하세요
  4. 최소한의 지침 작성: 격차를 해결하고 평가를 통과하기에 충분한 콘텐츠만 생성하세요
  5. 반복: 평가를 실행하고, 기준선과 비교하고, 개선하세요

이 접근 방식은 결코 실현되지 않을 수 있는 요구 사항을 예상하는 대신 실제 문제를 해결하도록 보장합니다.

Claude와 함께 스킬을 반복적으로 개발

가장 효과적인 스킬 개발 프로세스에는 Claude 자체가 포함됩니다. Claude의 한 인스턴스("Claude A")와 함께 작업하여 다른 인스턴스("Claude B")가 사용할 스킬을 만드세요. Claude A는 지침을 설계하고 개선하는 데 도움을 주고, Claude B는 실제 작업에서 테스트합니다. Claude 모델은 효과적인 에이전트 지침을 작성하는 방법과 에이전트에 필요한 정보를 모두 이해하기 때문에 이것이 작동합니다.

새 스킬 만들기:

  1. 스킬 없이 작업 완료: 일반 프롬프팅을 사용하여 Claude A와 함께 문제를 해결하세요. 작업하면서 자연스럽게 컨텍스트를 제공하고, 선호도를 설명하고, 절차적 지식을 공유하게 됩니다. 반복적으로 제공하는 정보에 주목하세요.
  2. 재사용 가능한 패턴 식별: 작업을 완료한 후 유사한 향후 작업에 유용할 컨텍스트를 식별하세요.
  3. Claude A에게 스킬 생성 요청: "방금 사용한 이 BigQuery 분석 패턴을 캡처하는 스킬을 만들어주세요. 테이블 스키마, 명명 규칙, 테스트 계정 필터링 규칙을 포함하세요."
  4. 간결성 검토: Claude A가 불필요한 설명을 추가하지 않았는지 확인하세요.
  5. 정보 아키텍처 개선: Claude A에게 콘텐츠를 더 효과적으로 구성하도록 요청하세요.
  6. 유사한 작업에서 테스트: 관련 사용 사례에서 Claude B(스킬이 로드된 새 인스턴스)와 함께 스킬을 사용하세요.
  7. 관찰을 기반으로 반복: Claude B가 어려움을 겪거나 무언가를 놓치면 구체적으로 Claude A로 돌아가세요.

효과적인 스킬을 위한 체크리스트

스킬을 공유하기 전에 다음을 확인하세요:

핵심 품질

  • 설명이 구체적이고 핵심 용어를 포함
  • 설명에 스킬이 무엇을 하는지와 언제 사용해야 하는지 모두 포함
  • SKILL.md 본문이 500줄 미만
  • 추가 세부 정보가 별도 파일에 있음 (필요한 경우)
  • 시간에 민감한 정보 없음 (또는 "이전 패턴" 섹션에 있음)
  • 전체에서 일관된 용어
  • 구체적이고 추상적이지 않은 예제
  • 파일 참조가 한 단계 깊이
  • 점진적 공개가 적절하게 사용됨
  • 워크플로우에 명확한 단계가 있음

코드 및 스크립트

  • 스크립트가 Claude에게 미루지 않고 문제를 해결
  • 오류 처리가 명시적이고 도움이 됨
  • "부두 상수" 없음 (모든 값이 정당화됨)
  • 필요한 패키지가 지침에 나열되고 사용 가능한 것으로 확인됨
  • 스크립트에 명확한 문서가 있음
  • Windows 스타일 경로 없음 (모두 슬래시 사용)
  • 중요한 작업에 대한 유효성 검사/확인 단계
  • 품질이 중요한 작업에 대한 피드백 루프 포함

테스트

  • 최소 3개의 평가 생성
  • Haiku, Sonnet, Opus로 테스트
  • 실제 사용 시나리오로 테스트
  • 팀 피드백 통합 (해당되는 경우)
